## 3. Findings

### F1 — The ADR-7 scoped path was unreachable from the CLI (High, fixed)

**What.** Half and quarter scopes need their own totals line. Those lines only
exist on the per-quarter/half **sub-games**, fetched by
`BetB2BScraper._enrich_with_subgames`, which returns immediately unless
`skin.features["subgames"]` is true. That flag:

- defaults to `False` in `config.py`,
- is set by **no** skin YAML (all 8 checked),
- had **no CLI flag** to turn it on.

So `_enrich_with_subgames` never executed. Every market a scrape extracts is
tagged `FULL_MATCH`, `_nearest_185_total(markets, "QUARTER_1")` finds nothing,
and `add()` skips the scope. **`scrape --ingest` could only ever emit 3 of the
9 ADR-7 scopes** — FULL_MATCH plus the two team totals — and reported success
while doing it.

**Why it stayed hidden.** The unit tests build market dicts with `scope` already
set, so they prove the exporter handles scoped markets correctly; nothing tested
that a scrape ever *produces* one. The gap is between two green components.

**Fix (`5f6e6db`).** `--subgames` on `scrape` and `poll`, wired through
`_scrape_one` (which both commands share), plus a stderr note when `--ingest`
runs without it saying exactly which scopes are going. Left opt-in rather than
implied by `--ingest`: it costs up to 8 extra `GetGameZip` requests per event,
which is the operator's call to make, not a side effect of asking for ingest.

### F2 — The documented CLI invocation silently did nothing (High, fixed)

`src/sites/betb2b/cli/main.py` had no `if __name__ == "__main__"` guard, so
`python -m src.sites.betb2b.cli.main scrape …` — the form AGENTS.md gives in 11
places — imported the module, ran nothing, and **exited 0**. A no-op that
returns success is worse than an error; it reads as a working command.
(`python -m src.sites.betb2b.cli` was the working spelling.)

Fixed by adding the guard, with a subprocess test over both spellings so a
silent no-op fails loudly. Verified the test fails without the guard.

### F3 — The ADR-8 fix had no regression test (High, fixed)

`20eda23` changed prediction-affecting behavior and added no test; the prior
session's own handoff flagged it as item 1.

**Fix (`fe79cdb`).** Tests that simulate what the engine's `s02_h2h_totals`
actually computes — `home_score + away_score` — and assert it equals the
scope's own quantity, across all 9 scopes:

| scope | expected sum | |
|---|---|---|
| FULL_MATCH | 229 | both teams, whole game |
| FIRST_HALF / SECOND_HALF | 121 / 108 | |
| QUARTER_1..4 | 64 / 57 / 55 / 53 | |
| HOME_TEAM_TOTAL | 107 | home only, away zeroed |
| AWAY_TEAM_TOTAL | 122 | away only, home zeroed |

Also pins the **orient-then-zero ordering**: when the feed lists the event's
away side as `team1`, the scores are swapped into the event's orientation
first. Zeroing before that swap keeps the wrong team's points under the right
key — a plausible-looking implementation that is silently wrong, and one no
structural check would catch.

**Verified by mutation, not by assertion:** deleting the zeroing fails 6 of
these tests; moving it before the orientation swap fails 3. A regression test
that has never been seen red is a claim, not a guard.

### F4 — Skin `features:` clobbered the family defaults (Medium, fixed)

`from_dict` merged `market_groups`, `market_types` and `sport_map` onto their
defaults, but assigned `features` wholesale. A skin naming one flag dropped the
other seven, leaving each to whatever default its call site passes to
`features.get(...)`. Latent today (no skin sets `features`) — and it stops being
latent the moment one does, which is exactly what per-skin ADR-7 enablement
needs. Fixed to merge, matching the tables beside it.

### F5 — Session 27's per-scope counts are not reproducible (Medium)

The Session 27 write-up reports run_3 producing **96 requests from 11 events**,
broken down as 11 FULL_MATCH + 11 FIRST_HALF + 11 SECOND_HALF + 44 quarters +
10 HOME_TEAM_TOTAL + 9 AWAY_TEAM_TOTAL.

Given F1, that scrape cannot have produced the 66 half/quarter requests. The
same document states, correctly, that all 4,115 odds snapshots were stored with
`scope='FULL_MATCH'` — which is precisely why those scopes are empty. The run
as described would have yielded **30** requests, not 96.

This matters beyond bookkeeping: the "HOME_TEAM_TOTAL asymmetry" that session
spent its investigation on (10 sent, 1 stored) was chased through code and
market data that were never the variable. **No conclusion about half/quarter
ingestion in that document rests on data that exists.** Re-run with `--subgames`
before trusting any of it.

## 7. Recommended Next Steps

The pattern behind F1, F2 and F5 is one thing, not three: **a component was
verified in isolation and the composition was never run.** The exporter's tests
proved it handles scoped markets; nothing proved a scrape produces them. The CLI
grammar tests proved the parser; nothing proved the module runs. Session 27
tested the exporter's output; nothing checked that the scrape feeding it had the
markets that output claimed.

Each of these was invisible to more unit tests. The cheap counter is one
end-to-end assertion per pipeline — a scrape whose result is fed to
`build_ingest_matches` and checked for the scope set it should contain.

That is the same lesson Session 27 recorded from the H2H bug ("structured
validation is not enough — simulate the downstream computation") one level up:
simulate the downstream **caller**, not just the downstream computation.

### F8 — The engine stores one prediction per `match_id` (High, upstream, not fixable here)

Querying `/api/predictions` after the run: **11 of 11 matches store exactly one
record, and it is the last scope sent.** Send order ends with the team totals,
so every match is stored as `AWAY_TEAM_TOTAL` — except the one event lacking an
away-total market, stored as `HOME_TEAM_TOTAL`. Observed 10 AWAY + 1 HOME. Zero
half or quarter records, despite 16 being accepted.

**This resolves F5's open question and Session 27's investigation.** The
"HOME_TEAM_TOTAL asymmetry" (10 sent, 1 stored, 9 AWAY) is this exact signature.
Not engine state from prior runs, not market data, not the exporter — all three
hypotheses that session tested, none of which was the variable.

**Consequence for ADR-7:** its premise — one match → N scoped predictions — is
delivered on the scraper side and now verified end-to-end, and is discarded on
arrival. 54 of 65 predictions are overwritten. Fixing it means keying the
engine's store by `(match_id, scope)`; that is a **scorewise-engine** change,
a different repo, and nothing on this side can work around it. Until then
`--subgames` is a pipeline-validation tool, not a production feed — it costs
~6 extra requests per event for data the engine drops.

Recorded as ADR-10; both blockers backlogged.
