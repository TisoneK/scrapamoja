## Findings

### Fixed

**F1 — `scorewise` console script never runs (High, fixed in `c8a4179`)**
- `pyproject.toml` `[project.scripts]` declared `scorewise = "src.main:cli"`, but
  `src/main.py:cli` is `async def`. setuptools' generated console-script wrapper
  calls the target synchronously, so `scorewise` would build a coroutine and
  discard it un-awaited — the command does nothing (RuntimeWarning only).
- **Fix:** added a sync `main()` wrapper that runs `asyncio.run(cli())` (the same
  pattern the `__main__` block already used) and repointed the entry point to
  `src.main:main`. `python -m src.main` behaviour is unchanged.
- *Note:* verified by static reasoning + `py_compile`; not exercised as an
  installed script (no 3.12 runtime). Standard, low-risk pattern.

### Flagged (not fixed — need a test runtime or are architectural)

**F2 — `datetime.utcnow()` used 1081× (Medium, backlog)**
- Deprecated in Python 3.12 (the project's own floor). The mechanical swap to
  `datetime.now(timezone.utc)` is **not** safe blind: `utcnow()` returns a *naive*
  datetime, `now(timezone.utc)` a *tz-aware* one — changing `.isoformat()` output
  (adds `+00:00`) and risking `TypeError` on naive/aware comparisons across the
  codebase. Needs a deliberate, test-backed migration. → `tasks/backlog.md`.

**F3 — Bare `except:` in async paths can swallow `CancelledError` (Medium, backlog)**
- 57 bare `except:` clauses in `src/`. On 3.12 `CancelledError` is a
  `BaseException`, so a bare `except:` (unlike `except Exception:`) *catches* it —
  directly at odds with the project's stated cancellation strategy. Many are in
  sync helpers (benign), but those inside async flows (e.g.
  `src/sites/github/flows/search_flow.py`) can eat cancellation. Audit and narrow
  to `except Exception:` where the intent is "swallow errors, not cancellation."
  Needs tests to change safely. → `tasks/backlog.md`.
- *Verified-correct counter-example:* `selector_mixin.py:63,99`
  `except (asyncio.CancelledError, Exception)` after an explicit `task.cancel()` +
  `await task` is the *right* pattern (swallowing the cancellation of a task you
  just cancelled). Not a bug.

**F4 — Fire-and-forget `asyncio.create_task(...)` without holding a reference (Low/Medium, backlog)**
- ~10+ sites create tasks whose handles are never stored (e.g.
  `src/sites/base/site_scraper.py:84` in construction,
  `src/sites/base/plugin_lifecycle.py:876,885`). Per CPython docs the event loop
  keeps only weak references, so such tasks can be garbage-collected mid-flight
  and their exceptions silently lost. Some are in shutdown/cleanup paths where the
  risk is lower. Audit; store references (or `await`) where correctness depends on
  completion. → `tasks/backlog.md`.
