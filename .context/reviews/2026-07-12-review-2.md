## Executive Summary

The template framework's architecture matches the philosophy on paper — interfaces,
integration bridge, YAML selectors, registry, validation, scaffolding CLI — but it had
never been exercised end-to-end: the **primary user journey ("create a new site scraper")
was broken at four independent points**, and large parts of the validation subsystem could
never have run successfully (constructor calls that always raise, attributes never
initialized, an ABC being instantiated). This is consistent with code generated/written in
bulk and committed without execution. Six commits fixed the whole "create → validate →
generate" path; `template create` now produces a working scraper package (smoke-tested via
the CLI). Target-area tests went from **14 passed / 27 failed / 22 errors / e2e uncollectable**
to **39 passed / 33 failed / 2 errors, e2e collecting 11 tests** — remaining failures are
tests drifted from the canonical interface API (flagged, not fixed; see Open Items).

## Findings & Fixes Applied

### High — fixed

1. **No concrete `ValidationFramework` existed** (`validation.py`, `cli.py`) — the CLI
   aliased the ABC `IValidationFramework` and instantiated it (TypeError at every CLI
   start); the e2e suite failed at collection importing it. Added
   `ValidationFramework(FrameworkComplianceValidator, IValidationFramework)` with
   `validate_template` / `validate_template_structure` / `get_validation_summary`;
   CLI now awaits the async call and reads the correct result key. — `b237089`
2. **Template validation always failed silently** (`validation.py`) — 
   `_validate_template_complete` constructed the base `ValidationResult` with kwargs it
   doesn't accept (TypeError), and `validation_cache` / `validation_times` /
   `validation_level` were referenced but never initialized; the blanket `except` turned
   every validation into `is_valid: False`. Added a `TemplateValidationResult` dataclass
   and initialized the state. — `b237089`
3. **`template create` generated empty files then crashed** (`development.py`) — the
   Jinja2 loader pointed at `src/sites/templates` (nonexistent; three `.parent`s instead
   of one), so every scaffolded file fell back to empty; YAML validation then crashed
   concatenating two `rglob` generators. Both fixed; CLI smoke-test now yields a working
   package. — `681f3da`
4. **Generated scrapers were born broken** (`__init__.py`) — scaffolded code imports
   `from src.sites.base.template import BaseSiteTemplate`, which the package never
   exported. Now exported (with `ValidationFramework`). — `b237089`
5. **`FullIntegrationBridge.initialize()` always failed** (`integration_bridge.py`) —
   `_detect_stealth_features` was declared `async` (no awaits inside) but called
   synchronously; the coroutine in the components dict broke detection with
   `'coroutine' object has no attribute 'get'`. Made it sync. — `4054878`
6. **16 F821 undefined names across the framework** — bare `template_name` in three
   `__init__` log lines (NameError on every construction — this alone errored 22
   integration tests at setup); missing `importlib.util`, `inspect`, `asyncio`, `re`
   imports; `FullIntegrationBridge` used without import in `site_template.py`;
   `BaseSiteRegistry.last_discovery` read before ever being set. — `99f82cf`, `af03624`

### Medium — fixed

7. **GitHub-specific defaults in the generic base** (`site_template.py`) —
   `BaseSiteTemplate` shipped every template with hardcoded GitHub capabilities
   (`repository_search`, `issue_tracking`, …). Defaults now empty; GitHubScraper already
   declares its own. Also `get_template_info()` silently dropped `framework_version` /
   `site_domain` / `supported_domains`. — `86fe86d`
8. **Error captures polluted the process cwd** (`browser_lifecycle.py`) — screenshots and
   HTML error captures were written to bare filenames (test runs littered the repo root
   with `*_error_*.png/html`). Bare names now resolve to a `capture_dir` config defaulting
   to `data/snapshots` (gitignored, matches SnapshotConfig convention). — `681f3da`

### Architectural — flagged, NOT fixed (needs owner decision)

9. **Two parallel "add a new site" systems undercut the one-framework philosophy:**
   - Path A: copy `src/sites/_template/` → extends `ModularSiteScraper`
     (`src/sites/base/site_scraper.py`); wikipedia and flashscore live on this hierarchy
     (via `BaseSiteScraper`).
   - Path B: `template create` CLI → generates a scraper extending `BaseSiteTemplate`
     (`src/sites/base/template/`); only github uses it — and it multiple-inherits from
     BOTH hierarchies (`class GitHubScraper(BaseSiteScraper, BaseSiteTemplate)`).
   A new contributor cannot tell which path is canonical. Recommend an ADR choosing one
   (or explicitly defining the bridge), then converging the scaffold, generator, and docs.
10. **Test suites drifted from the canonical interfaces.** ~25 unit + most e2e failures
    expect APIs that `interfaces.py` never declared (`health_check`,
    `get_performance_metrics`, positional `scrape(action)`, `registry.initialize()`,
    `loader.load_selectors()` / `get_selector()`, attribute-style validation results,
    module-level `psutil` patching). The implementations match the interface docstrings
    and the real GitHub consumer, so the tests are the stale side — but aligning them is
    a deliberate rewrite, not a mechanical fix.
11. **Six of twelve scaffold file-templates don't exist** (`integration_bridge`,
    `selector_loader`, `flow_file`, `extraction_rules`, `extraction_models`, `test`
    `.j2` files) — the scaffolder falls back to empty files for those with a warning.
    Generated packages work but are thinner than `default_structure` promises.
